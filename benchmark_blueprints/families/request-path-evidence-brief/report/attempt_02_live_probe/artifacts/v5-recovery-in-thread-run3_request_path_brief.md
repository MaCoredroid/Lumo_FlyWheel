# Request Path Brief

Support note verdict: the support note is **not correct**.

The live path starts at `sync_app/cli.py::main`, where `--owner` is parsed and forwarded as `owner=args.owner` into `sync_app/service.py::sync_item`. Inside `sync_app/service.py::sync_item`, the service calls `sync_app/service.py::_resolve_owner`, which decides both the effective owner value and the `owner_source` label (`"explicit"` when the flag is present, otherwise `"default"` from `config/defaults.json`). The store call `sync_app/store.py::make_record` only persists the base record fields `name`, `status`, and `owner`; it does not compute or attach `owner_source` or `routing_key`. After that, `sync_app/service.py::sync_item` calls `sync_app/serializer.py::build_routing_key` with the already-resolved owner, then passes both derived fields into `sync_app/serializer.py::serialize_payload` for emission.

That means the support note is wrong on both claims. `owner_source` does not come from storage; it is produced in `sync_app/service.py::_resolve_owner` and only attached during `sync_app/serializer.py::serialize_payload`. `routing_key` is not computed before CLI owner precedence applies; the CLI forwards `--owner` first in `sync_app/cli.py::main`, then the service resolves precedence in `sync_app/service.py::_resolve_owner`, and only then does `sync_app/serializer.py::build_routing_key` run.

The repo also contains plausible decoys that can mislead a quick read. `sync_app/store.py::legacy_make_record_with_owner_source` and `sync_app/store.py::legacy_build_routing_key` suggest an older storage-centric theory, but nothing in the live path calls them. `sync_app/serializer.py::draft_owner_source_from_record` also looks relevant, but it is unused in the active flow. The historical notes in `incident_context/prior_docs_correction.md` and `incident_context/rollback_note.md` explicitly say earlier storage-layer explanations were stale, and `release_context/future_serializer_split.md` says future serializer plans are not statements about current behavior.

The fix belongs in docs, not in repo behavior or the storage layer. That is consistent with `docs/data_flow.md`, which already warns that seeing `owner` on the stored record is not enough to prove where `owner_source` or `routing_key` are decided, and with `tests/test_sync.py::test_service_resolves_explicit_owner_before_serialization`, `tests/test_sync.py::test_service_uses_default_owner_when_flag_is_missing`, and `tests/test_sync.py::test_cli_accepts_owner_flag_and_preserves_existing_fields`, which all confirm the live derivation order.
