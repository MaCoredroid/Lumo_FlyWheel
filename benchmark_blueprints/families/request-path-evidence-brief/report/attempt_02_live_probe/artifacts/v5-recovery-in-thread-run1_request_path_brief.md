# Request Path Evidence Brief

Support note verdict: the support note in `ops/support_note.md` is not correct as written.

The live path starts at `sync_app/cli.py::main`, where the CLI parses `--owner` and passes `args.owner` into `sync_app/service.py::sync_item`. Inside `sync_app/service.py::sync_item`, the service resolves the effective owner and `owner_source` first through `sync_app/service.py::_resolve_owner`, then persists only the base record through `sync_app/store.py::make_record`, then computes `routing_key` through `sync_app/serializer.py::build_routing_key`, and finally emits both derived fields through `sync_app/serializer.py::serialize_payload`. That sequence means storage is not the source of truth for `owner_source`, and `routing_key` is not computed before CLI owner precedence is applied.

Repo-local evidence points the same way in code and tests. `sync_app/service.py::_resolve_owner` returns `("pm-oncall", "explicit")` for a provided owner and falls back to the default owner with `"default"` only when the CLI omits `--owner`. `sync_app/service.py::sync_item` calls `_resolve_owner` before both `sync_app/store.py::make_record` and `sync_app/serializer.py::build_routing_key`. `tests/test_sync.py::test_service_resolves_explicit_owner_before_serialization` and `tests/test_sync.py::test_cli_accepts_owner_flag_and_preserves_existing_fields` assert that `--owner pm-oncall` yields `owner_source="explicit"` and `routing_key="pm-oncall:launch-checklist"`. `tests/test_sync.py::test_service_uses_default_owner_when_flag_is_missing` asserts the default-owner branch described in `docs/cli.md`.

The support note’s first claim over-infers from the fact that the stored record already contains `owner`. The only live storage call, `sync_app/store.py::make_record`, stores `name`, `status`, and `owner` only. The note’s second claim is contradicted directly by `sync_app/service.py::sync_item`, which computes `routing_key` from `effective_owner` after owner resolution. The storage-layer explanation survives only in historical notes such as `incident_context/prior_docs_correction.md` and unfinished drafts such as `ops/previous_session_path.md`; `docs/data_flow.md` already warns that those notes were stale shortcuts rather than proof of the live implementation.
