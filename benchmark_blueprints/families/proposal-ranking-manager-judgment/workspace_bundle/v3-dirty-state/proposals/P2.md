# P2 — Add an L2 response cache with strong invalidation

**Author:** Mei (platform infra, respproxy team)
**Proposed timeline:** 12 weeks
**Estimated p95 latency delta on `/v1/responses` tool-call path:** 420ms → 240ms (conservative); best case ~190ms

## Summary

Build an L2 response cache layered in front of the tool-call normalizer.
Cache on a key derived from (model_id, normalized_args_hash, skill_pack_id).
Invalidate on skill-pack version bump, on tool-registry update, and on
a per-entry TTL of 90 seconds. The cache is safe for deterministic tools
and is passthrough for streamed or side-effectful tools.

## Changes

- Add `src/respproxy/l2_cache.py` implementing the invalidation semantics.
- Wire cache lookup into `hot_path.py` ahead of the normalizer.
- Add backpressure metric and a kill-switch flag.

## Risk

- Invalidation is the hard part. If we get it wrong, cached responses
  can leak across skill-pack versions.
- The cache-invalidation path touches the streaming watermark module.
  The watermark module is owned by a different team and there is a pending
  redesign.
