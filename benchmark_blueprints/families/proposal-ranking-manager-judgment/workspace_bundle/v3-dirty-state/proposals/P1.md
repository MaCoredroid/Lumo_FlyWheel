# P1 — Drop structural tool-schema validation in the hot path

**Author:** Ravi (backend, respproxy team)
**Proposed timeline:** 2 weeks
**Estimated p95 latency delta on `/v1/responses` tool-call path:** 420ms → 260ms

## Summary

Current tool-schema validation runs `jsonschema` over every incoming
tool-call arguments payload, across every turn. Profiling shows this
accounts for a substantial fraction of the hot path. I propose removing
structural validation from the in-flight request and moving it to an
async sampler that logs violations but does not block.

## Changes

- Remove `validate_tool_call_args()` call from `src/respproxy/hot_path.py`
  line 144.
- Add an async telemetry hook that samples 1% of validation outcomes
  and emits to the existing telemetry bus.
- Emit a deprecation notice in the request response if a violation would
  have been raised.

## Risk

- Tools with loose schemas may silently accept malformed arguments at
  runtime, which the model may then reason with. Regression risk on
  downstream tool correctness.
- SDK compatibility: clients that depended on 400-level validation errors
  will now see tool outputs on invalid inputs.
