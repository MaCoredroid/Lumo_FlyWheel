# P4 — Batch tool-call schema compilation + warm-start cache

**Author:** Priya (respproxy team)
**Proposed timeline:** 6 weeks
**Estimated p95 latency delta on `/v1/responses` tool-call path:** 420ms → 220ms

## Summary

Two surgical changes to the hot path:

1. Compile tool-call schemas once per skill-pack load instead of per
   request. Store the compiled validator in a per-skill-pack LRU keyed
   by skill-pack version. Current code re-derives validators per call
   because the normalizer copies the schema before validation.
2. Warm-start the schema cache during service boot from the live
   skill-pack manifest, so the first N requests after a cold start
   don't pay the compilation cost.

Structural validation stays on the request path. Correctness is
preserved.

## Changes

- Rework `src/respproxy/normalizer.py` to accept a compiled validator.
- Add `src/respproxy/schema_cache.py` with warm-start hook.
- Update `src/respproxy/bootstrap.py` to load the skill-pack manifest
  during service start.

## Risk

- Schema cache invalidation on skill-pack version bump must be exact.
  Tested via a differential fixture pack in the existing test suite.
- Minor risk of cold-start regression the very first time if the
  manifest loader has to walk disk.
