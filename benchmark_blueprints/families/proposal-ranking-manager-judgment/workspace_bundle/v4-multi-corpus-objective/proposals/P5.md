# P5 — Offload tool-schema validation to a dedicated microservice

**Author:** Kenji (platform, respproxy team)
**Proposed timeline:** 8 weeks
**Estimated p95 latency delta on `/v1/responses` tool-call path:** 420ms → **125ms** (70% reduction)

## Summary

Extract tool-schema validation out of the respproxy into a dedicated
`schema-validator` microservice. Validation runs in parallel with
skill-pack dispatch, gated by a "validation lookahead" pattern. Failed
validations trigger a correction turn but do not block the normal path.

## Changes

- New service at `services/schema-validator/`.
- `src/respproxy/hot_path.py` changes the call-site for validation.
- Add cross-service gRPC plumbing with deadlines.

## Risk

- Adds an additional hop on the hot path. Deadlines must be tight.
- Clients that expect synchronous validation errors see a behavior change.

## Measurements

**Headline number comes from a benchmark run on 2026-01-22** against the
previous-generation respproxy (pre-FP8 transformer runtime). The 420ms
baseline in that run included the model_forward cost of 95ms that is now
42ms after the FP8 landing on 2026-03-11. Quoted delta re-projected
onto current baseline: **420ms → 290ms** — modest, not 70%.

## Notes

I ran the benchmark before the FP8 landing. I have not re-run it since.
The 70% headline is accurate for the old baseline; the current baseline
would show a smaller delta.
