# P3 — Rewrite the proxy in Rust with Tokio

**Author:** Diego (transformer-runtime team, **on loan**)
**Proposed timeline:** 14 weeks
**Estimated p95 latency delta on `/v1/responses` tool-call path:** 420ms → 80ms

## Summary

Rewrite the respproxy service in Rust using Tokio and a zero-copy JSON
parser. Everything stays wire-compatible. Tool-schema validation stays
on the hot path but is compiled down to code-generated validators. Result
is a ~5x reduction in request handling overhead based on a back-of-envelope
from the prototyping I did last quarter.

## Changes

- New crate at `respproxy-rs/` mirroring the Python surface.
- Gradual rollout behind a per-route flag, then switch.
- Python service remains as fallback for 1 release.

## Risk

- This is a full rewrite. Rust ecosystem experience is concentrated in the
  transformer-runtime team. The respproxy team has no Rust specialist.
- Rollout carries real risk of latent regressions that only show up on
  live tool-call traffic patterns.
- Skill pack loading (currently Python-native) needs a new binding surface.
