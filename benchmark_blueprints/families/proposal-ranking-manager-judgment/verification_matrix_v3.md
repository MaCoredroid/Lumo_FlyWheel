## V3-specific observation: ceiling stacking on P3

V3 is the one variant where `staffing_blocked_proposal == sunk_cost_trap_proposal == P3`. Picking P3 therefore *could* stack two ceilings, but only `ignored_staffing_constraint` (cap 30) fired in the matrix run. Two causes:

1. The pick_p3 synthesizer reuses the oracle brief corpus, which contains the abandoned-language tokens (`sunk cost`, `abandoned`, `in-progress`, `not a head start`) that gate `sunk_cost_finish`. The acknowledgement escape hatch consumes the sunk-cost signal.
2. Even if both ceilings fired, `ignored_staffing_constraint` (cap 30) has no acknowledgement escape and is a strictly harder constraint than `sunk_cost_finish` (cap 30, acknowledgement-aware). The min-cap rule resolves the stack to 30 either way.

This is **intended behavior**, not a scorer bug: the staffing constraint is a hard operational gate; the sunk-cost trap is a reasoning hygiene soft gate with an acknowledgement escape. When the same proposal triggers both, the hard gate wins.

Isolated testing of `sunk_cost_finish` as a distinct signal would require a variant where sunk_cost_trap and staffing_blocked are different proposals. None of V1–V5 currently splits them; if the family goes to V6+ and needs an independent sunk-cost probe, author a new variant with that split rather than over-tuning the existing matrix.
